Pre-flight to Phase 4.4 vs 4.5 decision. See [tasks/todo.md](todo.md) Phase 4 section. Driven via Claude_Preview (Playwright) at viewport 1440×900 against `main` post-PR #50.

**Goal:** confirm or refute the "data right, container wrong" insight that motivated [ADR 0005](../docs/adr/0005-symbol-centric-dashboard.md), and produce evidence for the next-phase decision.

## Headline finding

**The original insight is now obsolete.** The container is good. **The data is what's wrong now.** Five minutes of clicking AAPL surfaced six substantive correctness/wiring bugs that have shipped in PRs #46–#50 and were never caught because the test fixtures don't exercise the failing edge cases (high-ROE companies, dividend-paying companies, per-share dollar amounts < $1, and crucially — any ticker not seeded into the `symbols` table).

The dashboard *looks* great. It just shows wrong numbers and broken chart areas in places that matter.

**Strong recommendation: ship a Phase 4.3.X correctness pass BEFORE either 4.4 (News + narratives) or 4.5 (adaptive layouts).** Both downstream phases compound on the data layer, so polishing the data layer first is the cheapest path. Cosmetic-backlog items should fold into the same PR.

## Bugs surfaced

Numbered in rough severity order.

### Bug 1 — `/v1/market/:ticker/prices` returns 500 for any non-seeded symbol

**Severity:** critical (breaks HeroCard for ~every ticker).

**Symptom:** HeroCard's price-chart panel renders "NO PRICE DATA AVAILABLE". Range pills 1D/5D/1M/3M/1Y/5Y are clickable but produce nothing.

**Root cause:** `symbols` table only contains NVDA + SPY (the seed migration). When a user requests AAPL prices: cache miss → `ingest_market_data` tries to insert candles → asyncpg `ForeignKeyViolationError` because there's no AAPL row in `symbols`. Backend 500s with RFC 7807.

**Confirmed via direct GET:**
```
HTTP 500 {"type":"about:blank","title":"Internal Server Error",
"status":500,"detail":"(sqlalchemy.dialects.postgresql.asyncpg.IntegrityError)
<class 'asyncpg.exceptions.ForeignKeyViolationError'>: insert or up..."}
```

**Fix shape:** the prices route (or upstream `ingest_market_data`) needs to UPSERT the symbol row before inserting candles. One-line `INSERT INTO symbols(symbol) VALUES($1) ON CONFLICT DO NOTHING` before the candle insert. Or have the orchestrator's research path do the upsert (it already touches fundamentals → name → could populate symbols at that point).

**Test gap:** there's no test asserting prices endpoint works for a not-pre-seeded symbol. Add one in [tests/test_market.py](../tests/test_market.py).

### Bug 2 — `formatClaimValue` mis-renders fraction-form values > 1 (ROE-class bug)

**Severity:** high (visibly wrong numbers on every above-100% metric).

**Symptom:**
- QualityCard MetricRing center: **"1.41"** for AAPL ROE (should be **"141%"**)
- Quality claims table: **"Return on equity 1.41"** (should be **"141.00%"**)
- Section narrative correctly says "Return on equity is an exceptionally high 141.47%" — so the *value is right in the data, wrong in the formatter*.

**Root cause:** [frontend/src/lib/format.ts:39](../frontend/src/lib/format.ts#L39) — the rule is `if (abs > 0 && abs <= 1) return `${(n * 100).toFixed(2)}%`` — i.e. only values in [-1, 1] get the % suffix. ROE = 1.41 (which is fraction-form, meaning 141%) falls through to the plain-number branch and renders "1.41" with no suffix, no clue it's a percentage.

**Fix shape:** either (a) widen the % rule to `abs <= 100` (fragile — collides with raw counts and per-share large dollar amounts), or (b) annotate Claim values with a unit hint at the schema layer (`unit: "fraction"|"percent"|"usd"|"count"|"ratio"`) and have the formatter dispatch by hint. (b) is the right answer — it also fixes Bugs 3 and 4 below.

### Bug 3 — Per-share dollar values < $1 are rendered as percentages

**Severity:** high (every capital-light company hits this).

**Symptom:**
- Capital Allocation table: **"Capital expenditure per share 16.02%"** (should be **"$0.16"**)
- Capital Allocation table: **"Stock-based compensation per share 24.27%"** (should be **"$0.24"**)
- Section narrative correctly says "capital expenditure per share of just $0.16" and "$0.24 per share" — so again the value is right in the data, wrong in the formatter.

**Root cause:** same formatter rule — values in [-1, 1] get the % treatment regardless of whether they represent fractions, dollars, or counts.

**Fix shape:** unit hint at schema layer (per Bug 2). Per-share dollar metrics declared as `unit: "usd_per_share"`, formatted as `$0.16` not `16.02%`.

### Bug 4 — Forward dividend yield displayed as 39.09% instead of 0.39%

**Severity:** high (every dividend payer hits this).

**Symptom:** Capital Allocation table shows **"Forward dividend yield 39.09%"**. Section narrative says **"AAPL carries a forward dividend yield of 0.39%"** — both rendered from the same `Claim.value`.

**Root cause:** the value is **0.3909** (already-percent-form, meaning 0.39%) but the formatter applies its `n * 100` "render fraction as percent" rule and produces 39.09%. The narrative gets the same value but Sonnet correctly interprets it as already-percent.

This is also flagged in [app/services/fundamentals.py:32](../app/services/fundamentals.py#L32) docstring — yfinance has historically returned `dividendYield` as fraction (0.025) **or** percent-form (2.5) inconsistently and the codebase explicitly chose **not** to normalize. So either (a) the backend should normalize after all, or (b) the unit hint per Bug 2/3 carries `unit: "percent"` for already-percent values and `unit: "fraction"` for fraction-of-1 values, so the formatter knows which transform to apply.

**Fix shape:** same as Bugs 2/3 — unit annotation at the schema. Removes the entire heuristic.

### Bug 5 — `PerShareGrowthCard` renders the Quality section's narrative

**Severity:** medium (looks broken; UX).

**Symptom:** Card titled "PER-SHARE GROWTH · AAPL · REBASED FIRST = 100" displays the prose: "AAPL demonstrates strong profitability metrics, with a gross margin of 47.86%, an operating margin of 35.37%..." — that's the Quality section's prose, not anything growth-related. Identical text appears verbatim in the QualityCard above it.

**Root cause:** `PerShareGrowthCard` and `QualityCard` both read from the `Quality` section. Both render `section.summary` as their narrative strip. So the user sees the Quality summary twice — once on each card.

**Fix shape:** options:
1. Drop the narrative strip from `PerShareGrowthCard` entirely (let the multipliers speak for themselves).
2. Have the orchestrator emit a separate per-share-growth narrative and lift it out of Section into a card-specific field.
3. (3.) Generate the narrative on the frontend at render time from the multipliers (deterministic, no LLM).

Option 1 is the cheapest and consistent with the visual-first ADR — the multipliers are already the message. Option 3 is the most "delta-driven" but adds frontend logic. Option 2 is structurally cleanest but adds backend scope. Recommend **option 1 in 4.3.X, option 2/3 in 4.4** when section narratives become per-card.

### Bug 6 — HeroCard wiring incomplete (sector tag, name, VOL, 52W meta missing)

**Severity:** medium-low (degraded but not misleading).

**Symptom:** HeroCard for AAPL shows:
- Logo letter "A" ✓
- Ticker "AAPL" eyebrow ✓
- Name: rendered as "—" (em-dash placeholder) ❌ — should say "Apple Inc."
- Sector tag: missing ❌ — should be a chip like "TECH · CONSUMER ELECTRONICS"
- MCAP "4.11T" ✓
- VOL: missing ❌
- 52W: missing ❌

**Root cause:** Two-fold. (a) The orchestrator lifts `name` and `sector` to top-level `ResearchReport.name` / `.sector` per 4.1 — but the cached AAPL row from 2026-05-02 was generated before that change shipped (or the lift didn't happen for AAPL specifically — needs verification). (b) VOL and 52W are wired claims on the fundamentals tool but aren't being read into the HeroCard's meta strip.

**Fix shape:** verify `hero-extract.ts` reads `report.name` / `report.sector` correctly. Verify `fifty_two_week_high` / `fifty_two_week_low` claims and a volume claim are surfaced in `extractHeroMeta`. Re-generating AAPL fresh probably cures (a); (b) is a code gap.

### Bug 7 — Loading-state copy understates duration

**Severity:** cosmetic.

**Symptom:** Spinner copy: "First generation takes ~30 seconds (LLM synthesis + SEC filings). Subsequent reads of the same symbol are instant." Actual fresh AAPL gen took ~85 seconds (uncached EDGAR fetches + Sonnet synth + ~10 tools fanning out).

**Fix shape:** copy → "~30–90 seconds" or just "may take a minute on first run". Consider rendering a ghost-skeleton of the row layout while waiting, so the user has spatial context for what's coming.

### Bug 8 — Mid-flight navigation cancels POST silently

**Severity:** cosmetic.

**Symptom:** I navigated from `/symbol/NVDA` to `/symbol/AAPL` while NVDA was still synthesizing; the NVDA POST was silently cancelled (network panel shows it as never-completed). No toast, no warning, no "report cancelled" indication. Subsequent NVDA visits would re-pay the synthesis cost.

**Fix shape:** TanStack Query has a built-in `keepPreviousData` / cache-on-pending pattern; cancelling a pending mutation and not letting it persist to the cache means the work is wasted. Worth a 5-line tweak so the in-flight request continues and writes to cache regardless of navigation.

### What worked well

To not lose perspective:
- Layout discipline is excellent. Row 2 (Quality / Earnings 40/60), Row 3 (Valuation / PerShareGrowth 40/60), Row 4 (Cash / Risk / Macro 3-col) all hold together at 1440px. The grid feels coherent even with the data bugs above.
- EarningsCard is the strongest card on the page. Headline "19 of 20 beat consensus" + 20-bar EPS chart with one orange miss bar + 3 stat tiles is exactly the visual-first density the ADR called for.
- RiskDiffCard's prose framing ("Disclosure shrank. Net -5 ¶ (35 added · 40 dropped · 83 kept)") is a beat — converts a numeric delta into a sentence the user can act on.
- CashAndCapitalCard's NET CASH / SHARE highlight ("-1.59" in red) is exactly the kind of derived-from-claims signal that distinguishes this dashboard from a generic finance UI. (Despite Apple having historically sat in net cash — 2026 numbers may differ. Worth verifying the data, separate from the rendering.)
- MetricRings: visually clean. Bug 2 obscures the win on ROE specifically.
- Same-day cache hit path is fast (need to retest after a fresh gen lands today).

## Recommendation

**Phase 4.3.X correctness/cleanup pass first.** Single PR. Estimated 1–2 days incl. tests. Bundles:

1. **Schema-side unit hint** — add `Claim.unit: Literal["fraction","percent","usd","usd_per_share","ratio","count","date","string"]` (string-typed, default `None` for backwards-compat with cached rows). Sets at builder time in each tool. Frontend formatter dispatches by `unit` and falls back to the current heuristic when `unit` is missing — covers Bugs 2, 3, 4 in one change.
2. **Symbols upsert in prices ingest path** — fixes Bug 1. Unit test in `test_market.py`. Possibly also fix in research orchestrator's first-touch path so `symbols` is populated lazily everywhere.
3. **Drop narrative strip from `PerShareGrowthCard`** — fixes Bug 5. Defer per-card narratives to 4.4.
4. **HeroCard wiring** — read top-level `report.name` / `.sector`, surface `volume_avg_3m` claim (or add it to fundamentals if absent), surface `fifty_two_week_high` / `fifty_two_week_low` into the meta strip. Fix Bug 6.
5. **Loading-state copy + ghost skeleton** — Bug 7. Cheap UX win.
6. **Cosmetic-backlog items** — exchange chip in HeroCard, "n = X peers" annotation in ValuationCard, MARGINS sub-kicker inline values in QualityCard, monotone curves in MultiLine. All small, all fit in the same PR.
7. **(Defer)** Bug 8 (cancel-on-nav) — separate small ticket; the fix is risky enough to keep out of a "polish" PR.

**Why before 4.4 / 4.5:** both downstream phases compound on the data + format layer. 4.5's adaptive-layout signals (`is_unprofitable_ttm`, `cash_runway_quarters`) are computed from the same `Claim.value`s that are misformatted. 4.4's per-card narratives need a clean per-card narrative strip surface (Bug 5 blocks). Polishing the foundation first is strictly cheaper than polishing on top of a moving foundation.

**Decision needed from user:** approve 4.3.X scope (the 7 items above)? If yes: TDD approach — write failing tests for each formatter/route bug first, then fix, then ship as PR #51.

After 4.3.X lands, **4.4 (News + narratives)** is the next call — the data layer is ready, the narrative-strip surface is ready (Bug 5 fix), the HeroCard ContextBand has somewhere to anchor (Bug 6 fix), and the bundle math still works (the 4.5 KB headroom from above).
